# 数据迁移方案

## 迁移策略

```
┌─────────────────────────────────────────────────────────────┐
│  迁移原则                                                    │
│                                                             │
│  1. 只迁移必要数据（用户信息 + 金币）                        │
│  2. 老系统保留，设为只读模式                                 │
│  3. 历史数据保留在老系统，用于对账                           │
│  4. 新系统独立运行，不影响老系统                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 迁移数据范围

### 需要迁移的数据

| 数据 | 老存储 | 新存储 | 说明 |
|------|--------|--------|------|
| 用户基本信息 | MongoDB | PostgreSQL | ID、账号、密码、昵称、头像 |
| 用户金币 | MongoDB/Redis | PostgreSQL/Redis | 金币余额 |
| 用户 Token | Redis | Redis | 登录状态 |

### 不迁移的数据（保留在老系统）

| 数据 | 说明 |
|------|------|
| 游戏记录 | 历史对局数据 |
| 账单历史 | 充值/消费记录 |
| 比赛记录 | 比赛历史 |
| 聊天记录 | 历史聊天 |
| 其他历史数据 | 用于对账查询 |

## 迁移流程

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  步骤1：数据导出                                            │
│  ├── 从 MongoDB 导出用户基本信息                            │
│  └── 从 Redis 导出金币余额                                  │
│                                                             │
│  步骤2：数据转换                                            │
│  ├── 转换用户数据格式                                       │
│  ├── 密码加密方式兼容                                       │
│  └── 生成数据对照表                                         │
│                                                             │
│  步骤3：数据导入                                            │
│  ├── 导入 PostgreSQL                                       │
│  └── 导入 Redis                                            │
│                                                             │
│  步骤4：数据验证                                            │
│  ├── 对比老库和新库数据                                     │
│  ├── 验证金币数量一致                                       │
│  └── 抽样验证用户信息                                       │
│                                                             │
│  步骤5：上线切换                                            │
│  ├── 部署新服务端                                          │
│  ├── 用户使用新客户端                                       │
│  └── 老系统设为只读                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 老系统改造

```yaml
改造内容:
  
  关闭功能:
    - 创建房间
    - 加入房间
    - 开始游戏
    - 下注/操作
    - 充值
  
  保留功能:
    - 登录（只读）
    - 查看金币余额
    - 查看账单历史
    - 查看游戏记录
    - 代理对账功能
```


## 数据格式对照

### 用户信息

```
老格式 (MongoDB):
{
  "_id": ObjectId("..."),
  "account": "user001",
  "pwdMD5": "xxx",
  "nickName": "玩家1",
  "icon": "http://xxx",
  "coin": 10000
}

新格式 (PostgreSQL):
Table: users
- id: BIGINT (主键)
- account: VARCHAR(64) (唯一)
- password_hash: VARCHAR(128)
- nickname: VARCHAR(64)
- avatar_url: VARCHAR(256)
- coin: DECIMAL(20,2)
- created_at: TIMESTAMP
- updated_at: TIMESTAMP
```

### 金币数据

```
老格式 (Redis):
Key: "user:coin:{userID}"
Value: "10000"

新格式 (Redis):
Key: "user:coin:{userID}"
Value: "10000"
(保持一致，便于兼容)
```

## 回滚方案

如果迁移失败：

1. 停止新服务
2. 清空新数据库
3. 恢复老服务正常模式
4. 通知用户使用老客户端

## 风险评估

| 风险 | 级别 | 应对措施 |
|------|------|----------|
| 数据丢失 | 高 | 迁移前完整备份 |
| 金币不一致 | 高 | 多次验证对比 |
| 密码不兼容 | 中 | 保持加密方式一致 |
| 迁移时间长 | 中 | 提前测试，预估时间 |
